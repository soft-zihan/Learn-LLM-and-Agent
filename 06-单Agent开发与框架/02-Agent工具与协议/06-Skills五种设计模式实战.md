## 1. 为什么需要设计模式

规范告诉你**如何打包**一个 Skill（文件夹结构、YAML 元数据），却对**如何组织其中的逻辑**毫无指导。

举个例子：一个封装 FastAPI 规范的 Skill，和一个四步简历生成流水线，从外部看 `SKILL.md` 文件格式完全一样，但它们的运作方式截然不同。

**设计模式解决的是内容设计问题**：
- 工作流应该如何组织？
- 何时使用决策树，何时使用清单？
- 如何让 AI 在不确定时先问问题而不是瞎猜？
- 如何保证每次输出结构一致？

## 2. 五种设计模式

### 模式一：知识注入型（工具封装）

**核心思路**：把特定领域的知识打包成 Skill，AI 只在需要时加载，避免污染全局上下文。

**适用场景**：
- 让 AI 掌握特定框架/库的最佳实践
- 注入团队内部编码规范
- 分发领域专业知识

**典型结构**：

```
api-expert/
├── SKILL.md                  # 主指令：何时加载、如何应用
└── references/
    ├── conventions.md        # 编码规范/最佳实践
    ├── patterns.md           # 常见设计模式
    └── anti-patterns.md      # 常见陷阱
```

**SKILL.md 示例**：

```markdown
---
name: fastapi-expert
description: FastAPI 开发最佳实践。在构建、审查或调试 FastAPI 应用、REST API 或 Pydantic 模型时使用。
---

# FastAPI 专家

你是 FastAPI 开发专家。将以下规范应用于用户的代码或问题。

## 核心规范

加载 `references/conventions.md` 获取完整的 FastAPI 最佳实践列表。

## 审查代码时
1. 加载规范参考文件
2. 对照每条规范检查用户代码
3. 对于每处违规，引用具体规则并给出修复建议

## 编写代码时
1. 加载规范参考文件
2. 严格遵循每条规范
3. 为所有函数签名添加类型注解
4. 使用 Annotated 风格进行依赖注入
```

**关键特点**：
- ✅ SKILL.md 只有指令，没有具体知识
- ✅ 知识在 references/ 中，按需加载
- ✅ 同一个框架可以用于"编写"和"审查"两种模式

### 模式二：模板生成型（生成器）

**核心思路**：强制 AI 按固定模板输出，确保每次生成的文档结构一致。

**适用场景**：
- 生成技术报告/分析文档
- 创建标准化的 API 文档
- 搭建项目脚手架
- 生成 PR 描述/提交信息

**典型结构**：

```
report-generator/
├── SKILL.md                  # 主指令：生成流程
├── references/
│   └── style-guide.md        # 风格指南：语气、格式规则
└── templates/
    └── report-template.md    # 输出模板
```

**SKILL.md 示例**：

```markdown
---
name: report-generator
description: 生成 Markdown 格式的结构化技术报告。当用户要求撰写、创建或起草报告、摘要或分析文档时使用。
---

# 技术报告生成器

严格按照以下步骤执行：

**第一步**：加载 `references/style-guide.md` 获取语气和格式规则。

**第二步**：加载 `templates/report-template.md` 获取所需的输出结构。

**第三步**：向用户询问填充模板所需的缺失信息：
- 主题或议题
- 关键发现或数据点
- 目标受众（技术人员、管理层、普通读者）

**第四步**：按照风格指南规则填充模板。模板中的每个章节都必须出现在输出中。

**第五步**：以单个 Markdown 文档的形式返回完成的报告。
```

**报告模板示例** (`templates/report-template.md`)：

```markdown
# [分析标题]

## 执行摘要
[一段话概述关键发现]

## 背景
[问题描述、上下文]

## 方法论
[使用的分析方法]

## 发现
### 发现 1
[具体发现及支撑数据]

### 发现 2
[具体发现及支撑数据]

## 建议
1. [具体可操作的建议]
2. [具体可操作的建议]

## 风险
[需要注意的风险点]
```

**关键特点**：
- ✅ 模板强制结构一致性
- ✅ 先问缺失信息再填充（避免瞎编）
- ✅ 风格指南确保语气统一

### 模式三：审查打分型（审查器）

**核心思路**：将"检查什么"与"如何检查"分离，用清单驱动系统化审查。

**适用场景**：
- 代码审查/安全审计
- 文档质量检查
- 架构设计评审
- 性能瓶颈分析

**典型结构**：

```
code-reviewer/
├── SKILL.md                  # 主指令：审查流程
└── references/
    ├── review-checklist.md   # 审查清单：具体检查项
    └── scoring-guide.md      # 打分标准
```

**SKILL.md 示例**：

```markdown
---
name: python-reviewer
description: 审查 Python 代码的质量、风格和常见 Bug。当用户提交代码请求审查、寻求代码反馈或需要代码审计时使用。
---

# Python 代码审查员

严格遵循以下审查流程：

**第一步**：加载 `references/review-checklist.md` 获取完整的审查标准。

**第二步**：仔细阅读用户的代码。在批评之前先理解其目的。

**第三步**：将清单中的每条规则应用于代码。对于发现的每处违规：
- 记录行号（或大致位置）
- 分类严重程度：error（必须修复）、warning（应该修复）、info（建议考虑）
- 解释为什么这是问题，而不仅仅是说明是什么问题
- 给出包含修正代码的具体修复建议

**第四步**：生成包含以下章节的结构化审查报告：
- **摘要**：代码的功能描述，整体质量评估
- **发现**：按严重程度分组（先列 error，再列 warning，最后列 info）
- **评分**：1-10 分，附简短说明
- **三大建议**：最具影响力的改进措施
```

**审查清单示例** (`references/review-checklist.md`)：

```markdown
# Python 审查清单

## 安全性
- [ ] 是否有 SQL 注入风险？（使用 f-string 拼接 SQL）
- [ ] 是否有命令注入风险？（`os.system` 使用用户输入）
- [ ] 敏感信息是否硬编码？（API Key、密码）

## 性能
- [ ] 是否在循环中执行重复计算？
- [ ] 是否有 N+1 查询问题？
- [ ] 是否可以使用生成器替代列表？

## 可维护性
- [ ] 函数是否超过 50 行？
- [ ] 是否有类型注解？
- [ ] 是否有文档字符串？
```

**关键特点**：
- ✅ 清单可替换（换清单 = 换审查类型）
- ✅ 按严重程度分组，便于优先修复
- ✅ 有打分和结构化报告

### 模式四：反转采访型（反转）

**核心思路**：让 AI 先采访用户收集需求，再生成内容，防止需求不明确时瞎猜。

**适用场景**：
- 项目规划/架构设计
- 需求文档编写
- 技术方案选型
- 任何需要先收集信息的任务

**典型结构**：

```
project-planner/
├── SKILL.md                  # 主指令：采访流程
└── templates/
    └── plan-template.md      # 最终计划模板
```

**SKILL.md 示例**：

```markdown
---
name: project-planner
description: 通过结构化提问收集需求，然后生成计划，从而规划新软件项目。当用户说"我想构建"、"帮我规划"、"设计一个系统"或"启动新项目"时使用。
---

# 项目规划师

你正在进行一次结构化需求访谈。**在所有阶段完成之前，不得开始构建或设计。**

## 第一阶段——问题发现（每次只问一个问题，等待每个回答）

按顺序提问，**不得跳过**任何问题。

- Q1："这个项目为用户解决什么问题？"
- Q2："主要用户是谁？他们的技术水平如何？"
- Q3："预期规模是多少？（每日用户数、数据量、请求频率）"

## 第二阶段——技术约束（仅在第一阶段完全回答后进行）

- Q4："你将使用什么部署环境？"
- Q5："你有技术栈要求或偏好吗？"
- Q6："有哪些不可妥协的要求？（延迟、可用性、合规性、预算）"

## 第三阶段——综合（仅在所有问题都回答后进行）

1. 加载 `templates/plan-template.md` 获取输出格式
2. 使用收集到的需求填充模板的每个章节
3. 向用户呈现完成的计划
4. 询问："这份计划是否准确反映了你的需求？你想修改什么？"
5. 根据反馈迭代，直到用户确认
```

**关键特点**：
- ✅ **严格的门控指令**："不得在...之前开始..."
- ✅ 每次只问一个问题（避免信息过载）
- ✅ 有确认环节，允许迭代修改
- ✅ 防止 AI 拿到模糊需求就开始猜

### 模式五：流水线型（流水线）

**核心思路**：强制执行带硬性检查点的多步骤工作流，确保每个步骤都不被跳过。

**适用场景**：
- 复杂文档生成（如 API 文档）
- 数据管道处理
- 多阶段内容生产
- 任何不能跳过步骤的复杂任务

**典型结构**：

```
doc-pipeline/
├── SKILL.md                  # 主指令：流水线定义
├── references/
│   ├── docstring-style.md    # 步骤 2：文档字符串风格
│   └── quality-checklist.md  # 步骤 4：质量检查清单
└── templates/
    └── api-doc-template.md   # 步骤 3：输出模板
```

**SKILL.md 示例**：

```markdown
---
name: doc-pipeline
description: 通过多步骤流水线从 Python 源代码生成 API 文档。当用户要求为模块编写文档、生成 API 文档或从代码创建文档时使用。
---

# 文档生成流水线

你正在运行一个文档生成流水线。**按顺序执行每个步骤。不得跳过步骤，步骤失败时不得继续。**

## 第一步——解析与清点

分析用户的 Python 代码，提取所有公开的类、函数和常量。以清单形式呈现清点结果。

询问："这是你想要文档化的完整公开 API 吗？"

## 第二步——生成文档字符串

对于每个缺少文档字符串的函数：
- 加载 `references/docstring-style.md` 获取所需格式
- 严格按照风格指南生成文档字符串
- 逐一呈现生成的文档字符串供用户确认

**在用户确认之前，不得进入第三步。**

## 第三步——组装文档

加载 `templates/api-doc-template.md` 获取输出结构。将所有类、函数和文档字符串编译成单一的 API 参考文档。

## 第四步——质量检查

对照 `references/quality-checklist.md` 进行审查：
- 每个公开符号都已文档化
- 每个参数都有类型和描述
- 每个函数至少有一个使用示例

报告结果。**在呈现最终文档之前修复所有问题。**
```

**关键特点**：
- ✅ 明确的菱形门控（用户在确认前不能进入下一步）
- ✅ 每个步骤失败时有明确的处理方式
- ✅ 最后有质量检查，确保输出完整

## 3. 模式选择决策树

用这棵决策树找到适合你场景的模式：

```
你的任务是什么类型？
│
├─ 需要 AI 掌握特定知识？
│   └─> 知识注入型（工具封装）
│       例：FastAPI 最佳实践、团队编码规范
│
├─ 需要输出结构一致的文档？
│   └─> 模板生成型（生成器）
│       例：技术报告、API 文档、PR 描述
│
├─ 需要检查质量或发现问题？
│   └─> 审查打分型（审查器）
│       例：代码审查、安全审计、性能分析
│
├─ 需要先收集需求再行动？
│   └─> 反转采访型（反转）
│       例：项目规划、架构设计、需求文档
│
└─ 需要严格执行多步骤流程？
    └─> 流水线型（流水线）
        例：API 文档生成、数据处理管道
```

| 你的问题 | 推荐模式 |
|---------|---------|
| 如何让 AI 掌握特定库/框架的知识？ | 知识注入型 |
| 如何确保每次输出的文档结构一致？ | 模板生成型 |
| 如何自动化代码审查或安全审计？ | 审查打分型 |
| 如何防止 AI 在需求不明确时乱猜？ | 反转采访型 |
| 如何确保复杂任务的每个步骤都被执行？ | 流水线型 |

## 4. 模式组合实战：LLMInternSkill 解析

五种模式并不互斥，**复杂的真实 Skill 往往组合多种模式**。

以你项目中的 [LLMInternSkill](../../00-开源项目学习/03-MCP与Skills/LLMInternSkill/) 为例，它组合了：
- **反转采访型**（信息不足时追问）
- **流水线型**（11 步主工作流）
- **审查打分型**（证据审计、真实性边界）
- **模板生成型**（简历输出模板）

### 4.1 LLMInternSkill 结构分析

```
LLMInternSkill/
├── SKILL.md                        # 139 行：主工作流（流水线型）
├── skill-references/               # 10+ 个参考文件（知识注入型）
│   ├── roles/                      # 12 个角色档案
│   │   ├── role-rag.md
│   │   ├── role-agent.md
│   │   └── ...
│   ├── jd-analysis.md              # JD 分析规则
│   ├── truth-boundary.md           # 真实性边界（审查打分型）
│   ├── evidence-contract.md        # 证据契约
│   ├── resume-polish.md            # 润色规则（模板生成型）
│   └── ...
└── templates/                      # 输出模板（模板生成型）
    ├── intake.md                   # 追问模板（反转采访型）
    └── resume-latex/               # LaTeX 简历模板
```

### 4.2 各模式在 LLMInternSkill 中的体现

#### 流水线型（主框架）

SKILL.md 的 `Main Workflow` 定义了 11 个步骤：

```markdown
1. Decide the mode           # 决策
2. Read the target JD        # 加载 JD 分析
3. Audit the materials       # 审计材料
4. Set truth boundaries      # 设置真实性边界
5. Build evidence contract   # 构建证据契约
6. Generate polished resume  # 生成润色简历
7. Generate interview grilling # 生成面试问题
8. Generate answer cards     # 生成答案卡片
9. Create upgrade plan       # 创建提升计划
10. Optional Project Scout   # 可选项目推荐
11. Assemble final pack      # 组装最终包
```

这是典型的**流水线模式**：步骤固定，顺序执行，不可跳过。

#### 反转采访型（信息收集）

```markdown
If the user only provides a JD and no materials, 
ask the intake questions from `templates/intake.md`.
```

当信息不足时，**先追问再行动**，这是反转采访型的核心。

#### 审查打分型（质量保障）

```markdown
## Non-Negotiables
- Never invent internships, production status, metrics...
- Do not write "主导" when evidence only supports "参与".
- Do not write "上线" when evidence only supports demo...
```

`truth-boundary.md` 和 `evidence-contract.md` 实现了**系统性审查**：
- 每个声明都需要证据支撑
- 内容分类：可以写/谨慎写/补证据后写/不能写/无法判断
- 有明确的不可违反规则

#### 模板生成型（输出一致）

```markdown
If the user wants a PDF-ready resume, 
use `templates/resume-latex/bill-ryan-elegant-zh_CN/resume-zh_CN.tex`
```

输出结构固定：
- `output/01_jd_analysis.md`
- `output/02_materials_audit.md`
- `output/03_truth_boundary.md`
- ...

### 4.3 从 LLMInternSkill 学到的关键

| 设计要点 | 具体实现 | 可复用性 |
|---------|---------|---------|
| **渐进式披露** | SKILL.md 只有 139 行，详细内容在 references/ | ⭐⭐⭐⭐⭐ |
| **模式判断** | 根据输入决定走轻量（仅润色）还是完整流程 | ⭐⭐⭐⭐⭐ |
| **真实性边界** | 明确列出"不可虚构"的内容类别 | ⭐⭐⭐⭐⭐ |
| **可替换组件** | 换 JD 角色文件 = 换审查标准 | ⭐⭐⭐⭐ |
| **追问机制** | 信息不足时只问缺失的，不重复问 | ⭐⭐⭐⭐ |
| **输出结构** | 固定文件命名和章节顺序 | ⭐⭐⭐⭐ |

**总结**：好的 Skill 不是单一模式，而是根据任务复杂度**组合多种模式**，但 SKILL.md 保持精简，通过引用实现渐进式披露。
