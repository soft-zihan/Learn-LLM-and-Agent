## 1. 垂域模型训练概述

### 1.1 为什么需要垂域模型？

通用大语言模型虽然在广泛任务上表现出色，但在垂直领域面临以下挑战：

- **专业知识不足**：缺乏领域特定的术语、概念和推理模式
- **准确性要求高**：医疗、法律、金融等领域对错误容忍度极低
- **合规与隐私**：需要遵守行业法规（HIPAA、GDPR、SOX 等）
- **实时性要求**：领域知识更新快，需要持续学习最新进展
- **可解释性**：高风险决策需要模型提供推理依据

**垂域 LLM 市场趋势（2025-2026）**：
- 全球垂域 LLM 市场从 2025 年的 $29 亿增长到 2033 年的 $187 亿（26% CAGR）
- 医疗、法律、金融、保险等受监管行业是主要应用场景
- 生产部署已从实验阶段转向规模化应用

### 1.2 垂域训练方法论

```
垂域模型训练流程
├── 1. 需求分析
│   ├── 领域范围定义
│   ├── 任务类型识别（问答、摘要、推理、生成）
│   └── 评估标准确定
│
├── 2. 数据策略
│   ├── 数据收集（公开数据集、企业内部数据）
│   ├── 数据清洗与去重
│   ├── 数据标注（专家标注、弱监督）
│   └── 数据混合比例优化
│
├── 3. 训练策略选择
│   ├── Continued Pretraining（领域继续预训练）
│   ├── Supervised Fine-tuning（监督微调）
│   ├── Parameter-Efficient Fine-tuning（参数高效微调）
│   └── RLHF/RLVR（强化学习对齐）
│
├── 4. 领域适配技术

└── 5. 部署与监控
    ├── 合规审查

    └── 知识更新机制
```

## 2. 医疗领域模型训练（概述）

### 2.1 医疗领域特点

医疗领域对模型准确性要求极高，主要应用场景包括：
- **临床决策支持**：辅助医生进行诊断和治疗方案制定
- **医学问答**：回答患者和医护人员的医学问题
- **病历分析**：从临床笔记中提取关键信息
- **药物研发**：加速药物发现过程

**主要挑战**：
- HIPAA 等隐私法规合规要求
- 医学术语和专业推理模式
- 错误容忍度极低
- 需要持续更新最新医学知识

### 2.2 常用数据集

| 数据集 | 规模 | 用途 |
|--------|------|------|
| MIMIC-IV | 40K+ 患者 | 临床决策支持 |
| PubMedQA | 500K QA | 医学问答系统 |
| MedQA | 40K+ 问题 | 医学考试评估 |
| MedMCQA | 194K 问题 | 多选择医学问答 |

### 2.3 关键技术点

**词汇表扩展**：医疗领域包含大量专业术语（疾病名称、药物名称、检查项目等），需要扩展模型词汇表以提升理解能力。

**Continued Pretraining**：使用大规模医学文献和临床笔记进行领域继续预训练，注入医学知识。

**隐私保护**：医疗数据需要进行严格的去标识化处理，移除患者姓名、身份证号、日期等 PHI（Protected Health Information）。

### 2.4 评估基准

主流医学评估基准包括：MedQA-USMLE、MedMCQA、PubMedQA、MMLU-Medicine 等，主要评估医学知识准确性和推理能力。

## 3. 法律领域模型训练（概述）

### 3.1 法律领域特点

法律领域需要模型具备严谨的逻辑推理能力和准确的法规引用能力，主要应用包括：
- **合同分析**：自动识别合同条款和潜在风险
- **案例检索**：查找相似案例和法律依据
- **法律咨询**：提供基础法律建议
- **文书生成**：辅助起草法律文档

**主要挑战**：
- 法律条文引用必须准确
- 案例推理需要严密的逻辑链
- 不同法域的法律体系差异大
- 法律语言高度专业化

### 3.2 常用数据集

| 数据集 | 规模 | 用途 |
|--------|------|------|
| Pile-of-Law | 270GB | 法律预训练 |
| LexGLUE | 多任务 | 法律模型评估 |
| CUAD | 500+ 合同 | 合同分析 |
| LEDGAR | 100K 条款 | 条款分类 |

### 3.3 关键技术点

**法律术语处理**：法律领域包含大量专业术语（原告、被告、管辖权、违约责任等），需要通过词汇表扩展或领域预训练来增强理解。

**法律推理训练**：通过构建"案例事实 → 争议焦点 → 法律适用 → 判决预测"的训练样本，培养模型的法律推理能力。

**合同条款识别**：使用信息抽取技术识别合同中的关键条款类型（付款、违约、保密、竞业禁止等）。

### 3.4 评估基准

LexGLUE 是主流法律 NLP 基准，包含 CaseHold（案例推理）、LEDGAR（条款分类）、ContractNLI（合同自然语言推理）等任务。

## 4. 金融领域模型训练（概述）

### 4.1 金融领域特点

金融领域对数值准确性和时效性要求极高，主要应用包括：
- **金融报告分析**：从财务报告中提取关键指标和趋势
- **投资研究**：辅助分析市场趋势和投资机会
- **风险评估**：识别金融风险和合规问题
- **智能客服**：回答客户金融问题

**主要挑战**：
- 数值推理精度要求高（财务计算不能出错）
- 金融知识更新快速（市场数据实时变化）
- 合规性要求严格（金融监管法规）
- 时间序列理解（历史数据趋势分析）

### 4.2 常用数据集

| 数据集 | 规模 | 用途 |
|--------|------|------|
| FinQA | 8,000+ QA | 金融推理（含计算） |
| FinanceBench | 500+ 问题 | 金融评估 |
| FPB | 10K 新闻 | 金融情感分析 |
| ConvFinQA | 3,000+ 对话 | 多轮金融问答 |

### 4.3 关键技术点

**数值推理增强**：金融领域需要模型具备准确的计算能力，通过构建"数据识别 → 公式列出 → 计算执行 → 答案输出"的链式推理训练样本来提升数值推理能力。

**时间序列理解**：金融数据具有强时间相关性，需要模型理解价格趋势、成交量变化、技术指标等时间序列模式。

**知识时效性**：金融市场变化快速，需要结合 RAG 技术引入最新市场数据和新闻。

### 4.4 评估基准

FinanceBench 是主流金融评估基准，测试模型在真实金融场景下的 QA 准确性、计算准确性和推理能力。

## 5. 代码领域模型训练（概述）

### 5.1 代码领域特点

代码领域模型需要具备代码理解、生成、修复和审查能力，主要应用包括：
- **代码补全**：智能代码自动补全
- **代码生成**：根据需求生成代码
- **代码审查**：识别潜在 bug 和安全漏洞
- **代码解释**：解释代码功能和逻辑

**主要挑战**：
- 多编程语言支持
- 代码语法和语义理解
- 长上下文代码理解
- 代码质量和安全性评估

### 5.2 常用数据集

| 数据集 | 规模 | 用途 |
|--------|------|------|
| The Stack v2 | 12TB | 代码预训练 |
| HumanEval | 164 问题 | 代码生成评估 |
| MBPP | 974 问题 | 代码生成评估 |
| SWE-bench | 2,000+ PR | 代码修复评估 |
| CodeSearchNet | 2M 函数 | 代码理解 |

### 5.3 关键技术点

**代码词汇表扩展**：代码包含大量特殊符号、关键字和标识符，需要扩展词汇表以更好处理代码 token。

**多目标任务训练**：
- 代码补全：给定代码前缀预测后续代码
- 代码解释：根据代码生成文档字符串
- 代码转换：在不同编程语言间转换

**函数级代码生成**：通过"文档字符串 + 测试用例 → 函数实现"的训练模式，培养模型生成可执行代码的能力。

### 5.4 评估基准

- **HumanEval**：评估 Python 代码生成能力（pass@1 指标）

```python
import re

class DataCleaningPipeline:
    """通用数据清洗管道"""
    
    def __init__(self, domain: str):
        self.domain = domain
        self.cleaners = self._init_cleaners()
    
    def clean(self, raw_data: list) -> list:
        """执行数据清洗"""
        cleaned_data = []
        
        for item in raw_data:
            # 1. 基础清洗
            item = self._basic_clean(item)
            
            # 2. 去重
            if self._is_duplicate(item):
                continue
            
            # 3. 质量过滤
            if not self._meets_quality(item):
                continue
            
            # 4. 格式标准化
            item = self._normalize_format(item)
            
            # 5. 领域特定清洗
            item = self._domain_specific_clean(item)
            
            cleaned_data.append(item)
        
        return cleaned_data
    
    def _basic_clean(self, item: dict) -> dict:
        """基础清洗"""
        # 移除特殊字符
        item['text'] = re.sub(r'[^\w\s.,;:!?()-]', '', item['text'])
        
        # 移除过长文本
        if len(item['text']) > 10000:
            item['text'] = item['text'][:10000]
        
        # 移除过短文本
        if len(item['text']) < 50:
            return None
        
        return item
    
    def _domain_specific_clean(self, item: dict) -> dict:
        """领域特定清洗"""
        if self.domain == "medical":
            # 医疗：去标识化
            item['text'] = deidentify_medical_text(item['text'])
        
        elif self.domain == "legal":
            # 法律：标准化引用格式
            item['text'] = normalize_legal_citations(item['text'])
        
        elif self.domain == "financial":
            # 金融：标准化数字格式
            item['text'] = normalize_financial_numbers(item['text'])
        
        elif self.domain == "code":
            # 代码：移除调试代码
```

```python
    扩展领域词汇表

        base_model_path: 基础模型路径

        vocab_size: 新增词表大小
    
    Returns:
        (tokenizer, model) 扩展后的模型
    """
    # 1. 加载基础模型
    tokenizer = AutoTokenizer.from_pretrained(base_model_path)
    model = AutoModelForCausalLM.from_pretrained(base_model_path)
    
    # 2. 训练 BPE 分词器
    from tokenizers import Tokenizer
    from tokenizers.models import BPE
    from tokenizers.trainers import BpeTrainer
    
    # 构建训练语料
    with open('domain_corpus.txt', 'w') as f:
        for text in domain_corpus:
            f.write(text + '\n')
    
    # 训练新分词器
    new_tokenizer = Tokenizer(BPE())
    trainer = BpeTrainer(
        vocab_size=vocab_size,
        special_tokens=["<unk>", "<s>", "</s>"]
    )
    new_tokenizer.train(['domain_corpus.txt'], trainer)
    
    # 3. 提取新 token
    new_tokens = new_tokenizer.get_vocab().keys()
    existing_tokens = set(tokenizer.vocab.keys())
    tokens_to_add = list(set(new_tokens) - existing_tokens)
    
    print(f"新增 token 数量：{len(tokens_to_add)}")
    
    # 4. 扩展词表
    tokenizer.add_tokens(tokens_to_add)
    model.resize_token_embeddings(len(tokenizer))
    
    return tokenizer, model
```

```python
    选择训练策略

    data_size = data_analysis['size']
```

### 7.2 常见陷阱

#### 数据陷阱

| 陷阱 | 描述 | 后果 | 解决方案 |
|------|------|------|----------|
| 数据泄漏 | 训练集包含测试集信息 | 评估指标虚高 | 严格划分数据集 |
| 数据偏见 | 数据分布不均衡 | 模型偏向某些类别 | 数据平衡、重采样 |
| 过时数据 | 使用过期知识 | 模型知识陈旧 | 定期更新数据 |
| 低质量标注 | 标注错误或噪声大 | 模型学习错误模式 | 专家审核、多标注者 |

#### 训练陷阱

| 陷阱 | 描述 | 后果 | 解决方案 |
|------|------|------|----------|
| 灾难性遗忘 | 微调后失去通用能力 | 模型泛化性下降 | 混合通用数据训练 |
| 过拟合 | 在训练集上表现过好 | 测试集性能差 | 正则化、早停 |
| 对齐税 | 对齐后能力下降 | 基准测试分数降低 | 平衡对齐与能力 |
| 奖励黑客 | 模型利用奖励函数缺陷 | 生成不合理但高分输出 | 多样化奖励信号 |

#### 部署陷阱

| 陷阱 | 描述 | 后果 | 解决方案 |
|------|------|------|----------|
| 合规风险 | 未通过行业认证 | 法律风险 | 提前合规审查 |
| 隐私泄露 | 模型泄露训练数据 | 违反隐私法规 | 差分隐私、数据去标识化 |
| 幻觉输出 | 生成不准确信息 | 用户信任丧失 | RAG、事实验证 |
| 性能不足 | 推理延迟过高 | 用户体验差 | 模型量化、缓存 |

### 7.3 成功案例概述

垂域模型训练的成功案例通常遵循以下模式：
- **大规模领域数据**：使用领域特定的大规模数据进行继续预训练
- **多阶段训练**：领域预训练 → 指令微调 → 对齐优化
- **严格评估**：使用领域专用基准进行全面评估
- **安全保障**：多层安全过滤和专家审核机制
- **受控部署**：在生产环境中逐步 rollout 并持续监控
